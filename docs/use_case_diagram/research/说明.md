# 科研数据支撑子系统用例图说明

## 1. 用例图概述

科研数据支撑子系统（Research Data Support Subsystem）用于支撑科研项目从立项申请、数据采集到科研成果提交与共享审批的完整业务流程。系统通过明确不同角色的职责划分，实现科研业务流程的规范化管理与数据可追溯。

本用例图重点描述科研人员与公园管理人员在科研业务线中的交互关系，以及科研项目与科研成果在提交与审批环节中的业务逻辑。

---

## 2. 参与者说明

### 2.1 Researcher（科研人员）

科研人员是科研数据支撑子系统的主要业务执行者，负责科研项目的申请、科研数据的采集与管理，以及科研成果的提交。

其核心目标是完成科研活动相关的数据与成果录入，并配合管理人员完成审批流程。

---

### 2.2 Park Manager（公园管理人员）

公园管理人员负责对科研相关业务进行管理与审批，确保科研活动符合管理规范和数据安全要求。

其主要职责包括科研项目立项审批以及科研成果共享权限的审批。

---

## 3. 用例说明

### 3.1 提交科研项目申请（Submit Project Application）

科研人员在系统中提交科研项目申请，填写项目基本信息，包括项目名称、研究领域、负责人、申请单位及项目计划时间等。

该用例是科研项目正式进入系统管理流程的起点。

**参与者**：Researcher
**用例结果**：生成待审批的科研项目信息

---

### 3.2 审批科研项目申请（Approve Project Application）

公园管理人员对科研人员提交的项目申请进行审核，根据管理规定决定是否批准立项。

该用例用于保障科研项目的合规性与合理性。

**参与者**：Park Manager
**关系说明**：

* 该用例被“提交科研项目申请”用例以 `<<include>>` 的方式引用，表示项目申请流程中必须包含审批环节。

---

### 3.3 管理科研数据采集（Manage Research Data Collection）

科研人员在科研项目执行过程中，对科研数据采集活动进行管理，包括数据的录入、调用系统已有监测数据以及对采集记录的维护。

该用例用于支撑科研过程中数据的持续积累和规范管理。

**参与者**：Researcher
**用例结果**：形成与科研项目关联的科研数据采集记录

---

### 3.4 提交科研成果（Submit Research Achievement）

科研项目完成或阶段性结题时，科研人员向系统提交科研成果信息，包括成果类型、成果名称、提交时间及共享权限设置等。

该用例用于实现科研成果的集中管理与归档。

**参与者**：Researcher
**用例结果**：生成待审批的科研成果记录

---

### 3.5 审批科研成果共享（Approve Achievement Sharing）

公园管理人员对科研人员提交的科研成果进行审核，重点审批成果的共享权限设置，确保保密成果仅对授权人员开放。

该用例用于保障科研成果的数据安全与合理共享。

**参与者**：Park Manager
**关系说明**：

* 该用例被“提交科研成果”用例以 `<<include>>` 的方式引用，表示科研成果提交必须经过审批后才能正式共享。

---

## 4. 用例关系说明

* **提交科研项目申请** `<<include>>` **审批科研项目申请**
  表示科研项目的申请流程中必须包含管理人员的审批环节。

* **提交科研成果** `<<include>>` **审批科研成果共享**
  表示科研成果在系统中发布或共享前，必须经过管理人员审批。

该用例图清晰地刻画了科研数据支撑子系统中科研项目与科研成果的关键业务流程，通过将“提交”和“审批”职责分离，体现了系统在科研管理、数据安全和权限控制方面的设计目标，为后续的类图、顺序图和数据库设计提供了明确的业务基础。
