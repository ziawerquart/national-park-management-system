用例图说明：

### 1. **系统边界（System Boundary）**

用例图的**系统边界**用来界定系统的范围。在本次用例图中，系统的名称是 **"Visitor Intelligent Management System"**，即“游客智能管理系统”。所有的用例和相关的功能都在该边界内，表示这些操作和服务属于系统的核心功能。用例图中的所有操作都是由系统提供的功能，而系统外部的角色与系统进行交互。

### 2. **角色（Actors）**

角色表示与系统交互的实体，它们通常是外部用户、系统管理员或其他系统。在这张用例图中，我们有以下几个角色：

* **Visitor（游客）**：游客是系统的主要用户，他们通过系统进行预约、入园、查看报告等操作。
* **SystemAdmin（系统管理员）**：系统管理员负责管理系统内的预约记录，核实游客信息，监控游客流量等功能。
* **ParkManager（公园管理人员）**：负责管理预约的批准，设定区域流量限制等任务。
* **SecurityPersonnel（执法人员）**：他们接收流量警报并采取措施，确保公园安全。

### 3. **用例（Use Cases）**

用例是系统提供给角色的一组服务或功能，它描述了系统如何响应某个角色的请求，并完成某项业务活动。在这张用例图中，定义了多个用例，下面是每个用例的详细描述：

* **Submit Reservation（提交预约）**：游客通过系统提交预约请求，选择入园时段、人数等信息。
* **Enter Park（入园）**：游客预约成功后，入园时需要进行身份核验。
* **View Statistics Report（查看统计报告）**：游客可以查看关于公园内游客流量、区域使用情况等的统计报告。
* **Receive Alert（接收预警）**：游客会在遇到安全隐患或流量超标时收到系统的警报通知。
* **Verify Reservation（核验预约）**：系统管理员需要核验游客的预约信息，确认预约是否有效。
* **Track Visitor Location（追踪游客位置）**：系统管理员通过系统追踪游客的实时位置，确保游客的安全并监控游客是否超出指定区域。
* **Manage Flow Control（管理流量控制）**：系统管理员可以监控当前公园内的游客流量，并做出必要的流量控制措施，例如限制流量。
* **Approve Reservations（批准预约）**：公园管理人员批准游客的预约请求，确认预约是否有效。

### 4. **用例之间的关系**

用例之间的关系表明了它们如何相互依赖，或者它们之间的执行顺序。主要有三种关系类型：`include`、`extend` 和普通的关联。

#### 4.1 **包含关系（<<include>>）**

包含关系表示一个用例必须包含另一个用例的执行。这意味着某个用例的执行必定会触发另一个用例的执行。我们在用例图中使用虚线箭头表示这种关系。

* **Submit Reservation（提交预约）** 包含 **Verify Reservation（核验预约）**：

  * 提交预约时，系统需要验证预约的有效性。核验预约是提交预约过程中不可或缺的部分，所以它是被包含在提交预约用例中的。
* **Track Visitor Location（追踪游客位置）** 包含 **Receive Alert（接收预警）**：

  * 当系统管理员追踪游客的位置时，系统会实时监控游客是否超出规定路线，若超出，则会触发警报。因此，接收警报是追踪游客位置时的一个必要步骤。
* **Track Visitor Location（追踪游客位置）** 包含 **Manage Flow Control（管理流量控制）**：

  * 在追踪游客位置的过程中，管理员还需要实时监控公园内的游客流量。如果游客过多，系统需要进行流量控制。因此，管理流量控制是追踪游客位置时的一个必要步骤。

#### 4.2 **扩展关系（<<extend>>）**

扩展关系表示在特定条件下，一个用例可以选择性地扩展另一个用例的行为。与包含关系不同，扩展关系是条件性的，只有在特定情况下才会发生。用虚线箭头表示这种关系。

* **Manage Flow Control（管理流量控制）** 扩展 **Receive Alert（接收预警）**：

  * 当公园的游客流量超过预设的阈值时，流量控制系统会触发警报，通知执法人员采取行动。这里的扩展关系意味着流量控制是在流量异常的情况下触发警报的。

### 5. **用例图整体结构**

* **系统边界**包裹着所有的用例，明确表示它们是系统的功能范围。
* **角色**位于系统边界外，表示它们与系统的交互，发起请求或接收响应。
* 用例之间通过关系箭头连接，标明它们的依赖关系。

### 总结

这张用例图的结构清晰地展示了“游客智能管理系统”的功能模块及其与不同角色之间的交互：

* 游客可以提交预约、入园、查看报告、接收预警。
* 系统管理员可以核验预约、追踪游客位置并管理流量控制。
* 公园管理人员主要负责批准预约。
* 执法人员主要接收预警并采取必要措施。
